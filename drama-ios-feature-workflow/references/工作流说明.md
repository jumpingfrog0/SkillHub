# iOS 新需求工作流

本工作流用于在 drama-ios 项目内启动和推进新的 iOS 业务需求。核心目标是把需求澄清、技术收敛、审批、编码和交付分开管理，保留关键决策追溯，同时避免每轮生成重复的大文档。

## 工作模式

只使用轻量快照评审模式。

历史快照只读，不得修改。

需求目录结构为：

```text
.huangdonghong/feature/<feature-id>/
  快照索引.md
  需求接收.md
  决策日志.md
  需求契约.md
  技术方案.md
  计划审批.md
  交付文档.md
```

`快照索引.md` 是需求入口和 checkpoint 时间线。其他文档按阶段逐步生成，不要求在启动时一次性填满。

## 旧版目录兼容

新建功能目录一律使用中文文件名。若已有目录仅包含旧版英文文件名，先按下表映射读取并继续原有流程；不得自动重命名、迁移或改写历史快照。

| 中文标准名 | 旧版英文名 |
|---|---|
| `快照索引.md` | `snapshot_index.md` |
| `需求接收.md` | `intake.md` |
| `决策日志.md` | `decision_log.md` |
| `需求契约.md` | `requirement_contract.md` |
| `技术方案.md` | `tech_plan.md` |
| `计划审批.md` | `approval.md` |
| `交付文档.md` | `handoff.md` |

## 模板位置

Skill 模板放在：

```text
.agents/skills/drama-ios-feature-workflow/assets/
```

模板文件包括：

```text
快照索引.md  # 需求入口、当前状态和 checkpoint 时间线
需求接收.md  # 原始需求、材料来源、目标业务范围和初始缺口
决策日志.md  # 需求确认、关键技术决策、需求变更和技术变更
需求契约.md  # 用户可见结果、业务规则、外部能力语义、设计/文案和验收标准
技术方案.md  # 唯一选定方案、需求语义映射、影响范围、关键数据流、风险和验证计划
计划审批.md  # 门禁检查、审批记录和审批基准
交付文档.md  # 交付摘要、验证结果、未验证项、已知风险和后续事项
```

编码过程状态由 Codex 会话维护，稳定决策写入 `决策日志.md`，最终交付写入 `交付文档.md`。

## 文档表达规则

需求事实使用标题、段落和编号列表表达。

checkbox 只用于状态跟踪、待确认项、任务完成情况和验证执行情况。

表格只用于材料来源、门禁检查、验证结果、阻塞项等多字段状态矩阵。

不得用 checkbox 表达功能范围、业务规则和最终技术结论。

## 需求与技术方案边界

需求契约确认“做什么”和“用户看到什么”，技术方案审批“如何实现”和“实现代价”。

`需求契约.md` 只允许记录：

- 功能范围、非目标、用户可见业务规则、状态/失败行为和验收标准。
- 设计稿、默认文案、跳转目标等设计与内容约束。
- 服务端、消息和配置能力的业务语义，以及该能力是否已确认。
- 已确认的多语言 key、默认中英文文案和使用场景。

除多语言 key 外，需求契约不得出现：客户端类、方法、回调、模块、文件、工程接入、数据流、算法、现有资源名、接口名、字段名、消息 subtype 或路由实现。

`技术方案.md` 必须将每项相关需求语义映射到精确协议/字段、资源、路由、内部模块和文件，并说明状态流、失败策略与验证方式。

## 阶段状态规则

阶段固定为：

- 需求接收
- 需求澄清
- 需求契约已确认
- 技术评审
- 待审批
- 已审批
- 编码中
- 验证完成

生成 `需求契约.md` 只表示需求契约草稿已生成，不代表需求澄清完成。

只要存在阻塞项、待确认项或用户 review 未完成，当前阶段必须保持为 `需求澄清`。

只有用户明确确认需求契约后，才允许把 `需求契约.md` 状态标记为 `已确认`，并把当前阶段推进为 `需求契约已确认`。

只有 `需求契约.md` 状态为 `已确认`，才允许进入技术评审并生成可推进的 `技术方案.md`。

`快照索引.md` 的 checkpoint 必须区分“需求契约草稿已生成”和“需求契约已确认”。

用户确认需求契约的固定口令为：

```text
CONFIRM_REQUIREMENT
```

收到 `CONFIRM_REQUIREMENT` 后，Agent 必须：

1. 读取 `快照索引.md`、`需求契约.md` 和 `决策日志.md`。
2. 检查契约是否完整表达业务规则、用户可见失败行为、边界、验收、多语言 key 和外部能力语义；并完成人工契约边界检查。
3. 如果仍有未决事项或混入内部实现细节，拒绝推进，保持当前阶段为 `需求澄清`，并输出剩余未决事项。内部实现细节移入技术评审，不要求用户通过需求确认审批实现路径。
4. 如果无未决事项，把 `需求契约.md` 状态标记为 `已确认`。
5. 把 `快照索引.md` 当前阶段推进为 `需求契约已确认`，并追加 checkpoint。
6. 在 `决策日志.md` 记录需求契约确认。
7. 进入技术评审，生成或更新 `技术方案.md`。

`CONFIRM_REQUIREMENT` 只表示需求契约已确认，不代表技术方案已审批，不允许进入编码，不允许生成通过状态的 `计划审批.md`。

## 启动规则

当用户要求启动 iOS 新需求工作流时，Agent 必须：

1. 根据需求名称生成可读的 `<feature-id>`，例如 `房间公屏@功能` 使用 `room_public_mention`。
2. 创建 `.huangdonghong/feature/<feature-id>/快照索引.md`。
3. 创建 `.huangdonghong/feature/<feature-id>/需求接收.md`。
4. 按用户输入的主要结构整理原始需求，记录材料来源、目标业务范围和初始缺口；不在本阶段选定内部模块。
5. 在 Codex 会话中完成第一轮需求澄清。
6. 不得进入编码阶段，除非 `计划审批.md` 已记录门禁通过，并且用户明确回复 `APPROVE_PLAN`。

如果已存在需求目录，Agent 必须先读取 `快照索引.md`，确认当前状态、审批基准和最近 checkpoint，再继续推进。

## 需求澄清规则

需求澄清阶段优先在 Codex 会话中完成，不为每轮问答创建完整文档。

只有稳定结论才写入文档。需求澄清阶段的稳定结论包括：

- 会影响用户可见范围、展示条件、状态、失败处理和验收的业务规则。
- 外部服务、消息或配置必须提供的业务能力及其语义。
- 会影响设计稿、文案、i18n key、跳转目标的内容约束。
- 会改变用户行为的兼容性或取舍。
- 阻塞项和解除条件。

以下内容属于技术评审，不得因代码阅读结果提前写入需求契约：

- 内部模块、类、方法、回调、文件和工程接入。
- 精确接口/字段/消息 subtype、资源名、路由实现和数据流。
- 缓存、排序、折叠等不改变用户需求的实现算法。

需求澄清阶段允许做最小代码静态阅读，目的仅限确认外部能力可用性、识别风险和形成待确认问题；不得据此选定实现方案。

如果技术细节本身会改变用户可见结果，应将其转换为业务问题后再确认，而不是把实现细节直接写入需求契约。

澄清过程中可以生成 `需求契约.md` 草稿。该文件只记录当前已确认内容和未决事项，不记录临时推理过程。

如果仍存在阻塞项，不得把 `需求契约.md` 标记为 `已确认`，不得生成可审批技术方案。阻塞项必须写入 `决策日志.md`，并明确解除条件。

## 多语言规则

新增文案必须确认多语言 key。

Agent 只在文档中给出建议 key、中文默认文案、英文默认文案和使用场景。

Agent 不修改 `lang.json`、`Localizable.strings` 等多语言产物。多语言文件由开发者通过自动化脚本更新。

多语言 key 未确认时属于阻塞项，不得进入技术评审、可审批技术方案或编码。

## 技术评审规则

技术评审阶段优先在 Codex 会话中完成，不为每轮方案讨论创建完整文档。

Agent 必须基于已确认的 `需求契约.md` 和代码静态阅读输出唯一选定方案，并生成 `技术方案.md`。方案必须包含从需求语义到精确协议/字段、资源、路由、内部模块、文件、状态流、失败策略和验证方式的映射。

`技术方案.md` 必须包含：

1. 目标和非目标。
2. 唯一选定方案。
3. 影响范围。
4. 关键数据流和状态流。
5. 风险和缓解方案。
6. 验证计划。

若技术方案新增 Objective-C/Swift 源码文件，必须把 `Drama.xcodeproj/project.pbxproj` 列入影响范围，用途仅限工程引用和 target membership。

技术评审发现任一情况时，必须暂停技术收敛并回到需求澄清：

- 外部能力无法满足已确认的业务语义。
- 候选方案会改变用户可见行为、状态保留、兼容性或验收结果。
- 存在必须由用户在产品体验、范围或成本之间选择的取舍。

此时更新需求契约和需求确认记录，并重新取得 `CONFIRM_REQUIREMENT`。仅影响实现且不改变已确认业务语义的调整，不回写需求契约，继续按计划审批规则处理。

进入审批前，`技术方案.md` 不得保留未收敛结论。无法收敛的内容必须回到 Codex 会话继续确认，并追加到 `决策日志.md`。

## 决策日志规则

`决策日志.md` 只追加稳定决策和变更，不写执行流水账。

必须分区记录：

- 需求确认记录：用户确认的业务规则、外部能力语义、设计/文案、跳转目标、阻塞项和多语言 key。
- 技术决策记录：技术方案中的实现映射与关键取舍，以及编码中发现审批方案需要变化的原因和用户确认结果。
- 阻塞跳过记录：用户明确跳过的阻塞项及允许的替代方案。

不得把普通任务进度、文件修改清单、临时搜索结果写入 `决策日志.md`。

## 计划审批门禁

生成 `计划审批.md` 前必须满足：

- 阻塞项为空，用户已明确跳过的阻塞项已记录在 `决策日志.md`。
- 必须确认问题为空。
- `需求契约.md` 已由用户明确确认，状态为 `已确认`。
- 需求契约边界检查通过：除多语言 key 外，未混入内部实现或精确协议/字段/资源标识。
- `需求契约.md` 已明确功能范围和验收标准。
- 新增文案的多语言 key 已确认。
- `技术方案.md` 已收敛为唯一选定方案。
- `技术方案.md` 已完成需求语义到精确协议、资源、内部范围、状态/失败处理和验证方式的映射。
- 技术评审发现的用户行为影响、外部能力不足或产品取舍，均已回到需求澄清并重新确认。
- 实施范围明确。
- 验证计划明确。

如果门禁不通过：

- 不得提示用户 `APPROVE_PLAN`。
- 不得进入编码。
- 必须在 `决策日志.md` 记录剩余阻塞项和解除条件。

## 计划审批

只有 `计划审批.md` 记录门禁通过后，才允许提示用户审批技术方案。

用户可回复：

```text
APPROVE_PLAN
```

用户要求调整方案时回复：

```text
REQUEST_CHANGES: <需要调整的内容>
```

用户允许跳过阻塞项时回复：

```text
SKIP_BLOCKER: <跳过的阻塞项和允许的替代方案>
```

收到 `REQUEST_CHANGES` 后，Agent 必须更新 `决策日志.md`，并根据变更内容更新 `需求契约.md`、`技术方案.md`、`计划审批.md`。

收到 `APPROVE_PLAN` 后，Agent 必须在 `计划审批.md` 记录审批结果和审批基准，再进入编码。

## 技术方案审批基准规则

`技术方案.md` 在审批前可以根据技术评审结果正常迭代。

收到 `APPROVE_PLAN` 后，`技术方案.md` 成为本次审批对应的技术方案基准，不再作为最终实现文档维护。

审批后不得按最终代码反向改写 `技术方案.md` 的方案正文。若需要标记状态，只允许追加简短说明，例如“本文件为本次 `APPROVE_PLAN` 对应的技术方案基准”。

编码中发现必须扩大影响范围或改变已审批方案时，Agent 必须暂停编码，说明变化原因和影响，并先更新 `决策日志.md`；必要时同步更新 `技术方案.md` 和 `计划审批.md` 形成新的审批基准，用户确认后才允许继续。

功能完成后的实际交付、最终实现与审批基准的差异、未验证项和风险统一写入 `交付文档.md`。

## 编码实施

进入编码后，Agent 必须遵守：

1. 只修改 `技术方案.md` 和 `计划审批.md` 列出的范围。
2. 如果需要扩大范围，先暂停并说明原因，用户确认后再更新 `决策日志.md`、`技术方案.md`、`计划审批.md`。
3. 不猜服务端接口、消息协议、路由规则、图片资源和 i18n key。
4. 不新增未审批的依赖。
5. 不维护单独的实现流水账文档。
6. 不静默忽略构建、测试、lint 和项目约定检查失败。

编码过程状态保留在 Codex 会话中。稳定决策变化写入 `决策日志.md`，最终交付写入 `交付文档.md`。

## 验证与交付

完成后必须生成或更新 `交付文档.md`，至少包含：

- 交付摘要。
- 实际改动范围。
- 偏离审批方案的内容。
- 验证结果。
- 未验证项。
- 已知风险。
- 后续事项。

如果无法运行某些验证，必须明确说明原因。

## 历史快照规则

历史功能快照保持只读，不迁移、不重写。
