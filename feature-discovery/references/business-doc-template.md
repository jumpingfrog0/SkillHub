# <功能名>业务梳理

## 背景与范围

- 说明本次梳理覆盖的功能、页面、入口、平台或模块。
- 说明不在本次梳理范围内的内容。
- 标注信息来源，例如代码路径、已有文档、配置或用户提供的上下文。

## 功能目标

- 说明该功能当前解决的业务问题。
- 说明目标用户、触发场景和业务价值。
- 如果目标只能从代码推断，明确标注为“推测”。

## 用户入口

- 列出用户可以进入该功能的入口。
- 标注入口对应的页面、按钮、跳转、路由、通知、深链、菜单或系统触发点。
- 记录入口是否受登录态、权限、版本、地区、配置、实验或开关影响。

## 用户路径

- 按用户视角描述主路径。
- 描述关键分支路径，例如成功、失败、取消、重试、空态、禁用态、未登录、无权限。
- 标注每个关键路径对应的证据来源。

## 核心业务规则

- 列出当前已确认的业务规则。
- 说明规则触发条件、判断依据和结果表现。
- 对无法确认的规则标注“推测”或放入“推测与待确认问题”。

## 状态、异常与边界条件

- 梳理加载态、空态、成功态、失败态、禁用态、降级态和异常态。
- 说明边界条件，例如网络异常、接口无数据、权限缺失、参数缺失、版本不兼容、配置关闭。
- 标注当前实现是否有明确处理；未发现处理时写“未发现”。

## 接口依赖

- 列出该功能依赖的接口、RPC、GraphQL、REST、SDK、系统 API 或本地服务。
- 说明请求触发时机、关键入参、关键出参和业务含义。
- 标注接口定义、调用点和响应处理位置。

## 配置、实验与开关

- 列出影响该功能展示、入口、逻辑、降级或策略的配置项。
- 列出相关实验、AB 分流、灰度策略、feature flag、服务端开关、本地开关或版本开关。
- 如果未发现配置、实验或开关，明确写“未发现”；如果需要产品、服务端或配置平台确认，写“待确认”。

## 文案、资源与埋点

- 列出用户可见文案、国际化 key、图片、动画、颜色、字体或其他资源。
- 列出曝光、点击、转化、错误、停留、结果等相关埋点。
- 标注资源和埋点的定义位置、触发时机和业务含义。
- 如果未发现资源或埋点，明确写“未发现”。

## 当前实现中的已确认事实

- 汇总可以被代码、配置或文档证明的事实。
- 每条事实尽量附证据，例如文件路径、类名、方法名、配置 key、接口名或埋点名。

## 推测与待确认问题

- 汇总无法从当前代码或文档中确认的信息。
- 将推测、风险、外部依赖和需要人工确认的问题分开表达。
- 不要把实现建议或改造方案写入本节。
