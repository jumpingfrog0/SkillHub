# Feature Discovery

使用这个 skill 探索已有功能，并记录当前实际存在的业务行为和技术实现。默认产物是结构化的现状理解；只有当用户明确要求写文档、沉淀文档、落地文档、生成文档时，才产出两份文档：

- `<feature-name>业务梳理.md`
- `<feature-name>技术实现梳理.md`

## 边界

- 不修改生产代码。
- 不写实现方案、方案选项、推荐方案、修改范围、任务拆分。
- 不把推测写成事实。明确区分已确认事实、推测行为、待确认问题。
- 重要结论优先给出证据，例如文件路径、类名、方法名、接口名、配置 key、实验名、feature flag、埋点名、资源标识。
- 配置、实验、feature flag、开关、埋点是必须显式检查的对象。如果未发现，记录“未发现”或“待确认”，不要静默省略。

## 项目约束

在探索和写文档前，先阅读并遵守当前仓库适用的项目约束，例如：

- `AGENTS.md`
- README
- 既有文档规范
- 验证限制
- 文档输出目录和命名规则

这个 skill 只定义通用方法论。正式探索文档根目录固定使用 `.huangdonghong/docs/`。写文档时如果该目录不存在，创建该目录。

## 文档目录规则

所有项目统一使用以下目录结构：

```text
.huangdonghong/docs/<feature-slug>/
```

- `<feature-slug>` 必须来自功能标识，禁止使用 `feature-discovery` 作为业务文档子目录名。
- 功能标识从用户请求的功能名、业务名、模块能力名中提取。
- 跨模块能力使用能力名作为功能标识，不强行归入单个源码模块。
- 功能标识使用小写短横线命名；中文功能名可转换为稳定英文缩写，例如 `RN侧SVGA和VAP能力` 使用 `rn-svga-vap`。
- 写文件前，先在对话中明确最终目录和文件名。
- 如果 `.huangdonghong/docs/<feature-slug>/` 下已经存在同功能的业务梳理或技术实现文档，默认更新已有文档，不新增重复文档。

## 文档信源规则

在探索功能代码前，先在 `.huangdonghong/docs/` 中搜索并阅读与功能名、模块名、入口名相关的现有业务文档和技术实现文档。

已有文档是待验证的现状理解，不是最终事实来源。最终结论仍需通过代码、配置、资源、文案、埋点、接口、数据流证据确认。

默认不要读取以下目录作为功能探索事实来源：

- `.huangdonghong/feature`
- `.huangdonghong/report`
- `.huangdonghong/reports`

这些目录可能包含实验性开发内容或历史汇报总结。只有用户明确要求参考时才读取，并在输出中标注为参考材料，不能直接当作当前功能事实。

## 工作流

1. 从用户请求中确认探索范围：
   - 功能名称
   - 目标平台和模块
   - 已知入口、页面、接口、命令、屏幕、用户动作
   - 用户是只需要探索回复，还是明确要求写入文档
2. 在下结论和选择文档路径前，先阅读项目约束。
3. 搜索并阅读 `.huangdonghong/docs/` 中的相关已有文档，建立待验证的文档认知。
4. 定位入口：
   - 路由、页面、控制器、Fragment、Activity、组件、命令、handler、service、API
   - 用户动作、回调、delegate、observer、listener、生命周期方法、reducer
5. 追踪当前行为：
   - 用户路径
   - 业务规则
   - 状态变化
   - API 和数据流
   - 持久化和缓存行为
   - 配置、实验、开关、降级、资源、文案、埋点依赖
6. 建立技术实现地图：
   - 关键文件
   - 关键类、结构体、方法、函数、组件
   - 调用链路
   - 数据模型和响应处理
   - 错误、空态、加载、禁用、兜底、边界状态
7. 归纳发现：
   - 区分已确认事实和推测行为
   - 列出缺失证据和待确认问题
   - 记录文档覆盖情况：已有文档已覆盖、代码确认后需修正、代码中发现但文档未覆盖
   - 避免提出改动建议；用户另行要求实现工作时，已经超出本 skill 的默认范围
8. 如果要写文档，先读取 `references/` 中的模板，并按项目约束填充：
   - `references/business-doc-template.md`
   - `references/technical-implementation-template.md`

## 输出规则

只回复、不写文件时，保持以下结构：

1. 业务梳理
2. 技术实现梳理
3. 已确认事实
4. 文档缺口
5. 待确认问题

写文件时：

- 除非用户另有要求，只写两份探索文档。
- 文件名使用功能名加固定后缀：`业务梳理.md` 和 `技术实现梳理.md`。
- 文档根目录固定为 `.huangdonghong/docs/<feature-slug>/`。
- 写入前必须向用户明确最终目录和文件名。
- 作者按项目约束填写；项目未指定作者时，不要编造作者。
- 每份文档必须同时包含“摘要版表达”和“完整归档”。
- 摘要版放在文档前部，保留聊天回复的清晰结构：关键结论、主链路、已确认事实、文档缺口、待确认问题。
- 完整归档放在摘要后部，用于补充证据、边界、配置、资源、埋点、异常、技术链路。
- 关键结论必须尽量包含证据。
- 对缺失的配置、实验、开关、埋点、资源、文案证据进行显式标注。

## 完整归档压缩规则

- 模板是检查清单，不是必须逐章填满的固定大纲。
- 同一事实只在一个主要章节展开，其他章节只引用结论。
- “未发现”类内容集中归并，不在多个章节重复铺开。
- 业务文档只展开业务范围、用户路径、业务规则、配置/资源/埋点、已确认事实、待确认问题。
- 技术文档只展开技术范围、入口调用链、关键文件职责、核心实现、数据流/状态流、配置与边界处理、待确认问题。
- 调用链优先使用短代码块表达，避免自然语言多次复述。
- 关键文件只列参与链路的文件，不做目录盘点。
- 章节没有实质内容时保留一行结论，不扩写解释。
- “完整归档”表示证据完整、结构可追溯，不表示模板所有栏目逐项扩写。
